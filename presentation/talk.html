Card 5

@soql — natural language to SOQL

3:00 – 4:30
Slide
@soql — natural language to SOQL
  • Grounded in your org's real schema.
  • Validated against the Salesforce planner before you see it.
  • Never executes — you copy and run it yourself.

Say (before)

"First one. SOQL Generator. You describe what you want, it gives you a query using real fields from your org — no hallucinated columns. And it never runs the query. You copy it into Workbench yourself."

Do (live)

  1. ⌘⇧K → soql → Enter.
  2. Type: "all open cases for Acme assigned to me this week".
  3. Wait for output. Highlight the field names with the cursor.

Say (after)

"Notice the field names — those came from the Case describe, not from Claude guessing. If you have a custom field Resolution_Path__c, it'll use that. If you don't, it won't invent one."
Card 6

@debug — the strongest demo

4:30 – 6:30
Slide
@debug — a flow debugger that actually reads your flow
  • Pulls the Flow metadata via Tooling API.
  • Returns Summary, Root cause, Suggested fix.
  • Read-only. Nothing in your flow changes.

Say (before)

"This is the one I use most. You're debugging a flow, you have the Debug Details panel open, you can see what happened but not why. Skipper grabs the flow's metadata from the Tooling API, sends both to Claude, and you get summary, root cause, suggested fix."

Do (live — prepped beforehand)

  1. Open the failing flow you prepared.
  2. Open the saved debug-output text file → copy.
  3. ⌘⇧K → "Debug this flow" → paste → Analyze.
  4. Read Summary aloud as it appears.
  5. Read Root cause aloud.
  6. Read step 1 of Suggested fix.

Say (after)

"Read-only. It doesn't touch the flow. It just tells you where to look."
Fallback If it hangs more than 15 seconds, switch to the pre-recorded clip. Don't apologize — "here's one I prepared earlier."
Card 7

@ask — the wow finish

6:30 – 8:30
Slide
@ask — screenshot + agentic tools
  • Takes a screenshot of the page.
  • Runs read-only tools to ground its answer.
  • SOQL, sObject describes, Apex search, field history.

Say (before)

"You're on any record page, something looks off, you don't know why. Hit Skipper, type ask, ask your question. It takes a screenshot, then runs read-only tools — SOQL queries, sObject describes, Apex code search, field history — until it can give you a real answer. Watch the activity panel while it works."

Do (live)

  1. Open a record page with something interesting (validation rule, oddly-populated field, permission quirk).
  2. ⌘⇧K → ask → ask the question.
  3. Narrate tool calls as they appear: "describing the object… searching Apex for that field name… checking field history on the record…"
  4. Read the final answer aloud.

Say (after)

"Strictly read-only. No DML, no anonymous Apex, no metadata changes. The screenshot and tool results go to Anthropic under your own API key."
Fallback Pre-record a 45-second screencap of this exact flow. Use it if the live one stalls past 25 seconds.

Pre-flight (morning of)

  1. Open Skipper once in the demo org before you walk on — first load fetches caches; don't pay that on slide 4.
  2. Test your Anthropic key with a throwaway @soql query right before you start. If the key is rate-limited or out of credit, the whole second half dies silently.
  3. Have the @debug fixture flow open in a second tab, plus the saved debug-output text file ready to copy.
  4. Pick the @ask record page the day before and write down the exact question. Don't improvise on stage.
