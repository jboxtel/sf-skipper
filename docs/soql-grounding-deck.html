Making @soql understand your org

What we built on feature/soql-grounding — and why it matters for every customer org the extension runs in.

Two-track deck: business case first, technical depth second.

The problem

The same English prompt produces different correct SOQL in different orgs.

An admin types "cancelled flight with assignments". The right answer depends entirely on how their org models flights:

Org"Flight" lives on…Correct SOQL
Airline using Service CloudAsset with record type FlightAssignmentFROM Asset WHERE RecordType.DeveloperName = 'FlightAssignment'
Charter operatorCustom object Flight__cFROM Flight__c
Travel agencyProduct2 with Family = 'Flight'FROM Product2 WHERE Family = 'Flight'
Data Cloud installMisleading Flight__dlm (DMO, no operational data)NOT FROM Flight__dlm — that's the lure

Claude has no training-data prior for "what does this org mean by flight?" The signal lives in the org's metadata. Grounding is the act of surfacing the right slice of that metadata into the prompt.

Refresher

What @soql is, in one paragraph.

A natural-language → SOQL feature inside Salesforce Commander. The user types "show me cancelled cases from email last week", the palette returns runnable SOQL grounded against their actual org's schema, ready to paste into Workbench, Developer Console, or an Apex class.

Without grounding

Claude guesses from training-data priors. Works for vanilla orgs, falls apart on any customization. Invents fields, picks the wrong object, hallucinates picklist values.

With grounding

Before each call, we pull the org's real schema — objects, fields, record types, picklist values, record counts — and hand the relevant slice to Claude alongside the prompt.

Concrete example

Same prompt. Same model. Different result.

Prompt: "Claim line items where the affected flight had revenue over 1000 euros" — against an airline Service Cloud org.

BEFORE

SELECT Id, Name, AffectedFlight__c,
       Amount__c, Claim__c, FlightRoute__c
FROM ClaimLineItem__c
WHERE AffectedFlight__c != null
  AND Amount__c > 1000

Claude saw AffectedFlight__c as a reference but didn't connect that Asset.RevenueEUR__c was the right field to filter on. Defaulted to a sibling field on ClaimLineItem__c — wrong data.

AFTER

SELECT Id, Name, AffectedFlight__c,
       AffectedFlight__r.RevenueEUR__c,
       Amount__c, Claim__c
FROM ClaimLineItem__c
WHERE AffectedFlight__r.RevenueEUR__c > 1000

The grounding pipeline now surfaces the related Asset object's schema alongside ClaimLineItem__c, with the dot-walk syntax (AffectedFlight__r.<field>) inline. Claude uses it correctly.

Measured results

How the current grounding performs.

Measured against 19 prompts spanning 6 fixtures (1 real-org snapshot, 5 synthetic shape isolations):

Sonnet 4.6 18/19 95% — only the synthetic picklist-value gap remains
Haiku 4.5 (production default) 17/19 ~89% — strong; remaining gaps are model-comprehension flavored, not grounding-shaped

Where the wins came from

  • Data Cloud filter — excludes __dlm/__cio/__dll objects that share labels with real ones (high-record-count lures).
  • Record-type-aware scoring — finds "claim", "flight", "settlement" attached to standard objects via their record types.
  • Lookup-chain grounding — fetches related-object schemas and surfaces dot-walk syntax inline.
  • Picker-fallback robustness — model can refuse cleanly instead of producing a parser error.
  • SOQL grammar nudges — closes specific weaknesses (no JOINs, semi-join syntax, dot-walks).
Why this matters

What it means for Salesforce Commander users.

Works on real customer orgs, not just demos

The measurement substrate is a real customer-org snapshot. Every grounding change is tested against shapes that actually exist in the wild — multi-record-type Cases, label hijacks (Asset = "Booking"), zero-record metadata traps, German legal terminology, six-orders-of-magnitude record count spreads.

Adapts to per-org customization

No customer-specific configuration. When a new org's metadata is pulled, the same pipeline ingests its record types, custom fields, picklists, counts — and adapts. An aviation customer's "flight" maps differently than an insurance customer's "claim". Same code, different ground.

Won't hallucinate the wrong system

Data Cloud DMOs (__dlm) frequently shadow operational objects with identical labels. The filter ensures @soql always points at live operational data — never at the analytics mirror that won't reflect today's writes.

Regression-protected, evolving

Every grounding change runs through an automated eval before it ships. New customer-org snapshots can be added as fixtures — each one becomes a permanent test case. Model upgrades are validated, not just hoped for.

Section 2

Technical depth

How the pipeline is structured, what data it pulls, and how the eval harness measures it.

Architecture

The grounding pipeline, end-to-end.

prompt: "cancelled flight with assignments" | v [1] describeGlobal (cached) objects.js — every queryable object's api name + label | v [2] loadRecordTypes (cached) soql.js — one SOQL: SELECT FROM RecordType | v [3] Lexical + record-type scoring soql.js — score api name, label, record-type names | against the prompt's tokens v [4] Top 3 candidates | v [5] fetchDescribe + fetchCount soql.js — parallel, per-candidate | also fetches up to 3 lookup-target describes v [6] buildUserMessage soql.js — schema + counts + record types + dot-walk hints | v [7] callClaude shared.js — system prompt + user prompt | v [8] parseSoqlResponse + validateSoql soql.js — /query?explain validates against the planner | retry on error with cumulative failure context v result: { soql, objectName, explanation }

Steps [1]–[4] are pure metadata. Step [5] is parallel HTTP. Step [8] is the self-correction loop — the Salesforce planner is the authority on grammar.

What we ground on

Five sources of grounding signal, each one HTTP call away.

SignalAPI endpointUsed for
describeGlobalGET /sobjectsObject name + label lexical scoring; the candidate universe.
Per-object describeGET /sobjects/<Name>/describeFields, picklist values, references, relationship names. Fetched lazily for top candidates and their lookup targets.
Record typesSELECT FROM RecordTypeBusiness-domain terms encoded on generic standard objects ("Claim" on Case, "Flight Assignment" on Asset).
Record countsSELECT COUNT() FROM <Name>Disambiguating near-duplicate objects (operational vs legacy, real vs DMO lure).
Reference field metadata(part of describe)Lookup-chain dot-walks: XXX__c + relationshipName → XXX__r.<field>.

All of these existed in the org's metadata before this work — the change was teaching @soql to fetch and surface them in the right combinations.

Measurement

The eval harness, in two paragraphs.

test/soql-eval.js drives Playwright to inject soql.js and its dependencies into a blank page. Salesforce REST endpoints (describeGlobal, describe, COUNT(), /query?explain, RecordType) are stubbed against JSON fixtures. Claude calls route through the real Anthropic API via page.exposeFunction — so the model actually grounds against the fixture's metadata, with whatever model the runner is configured for.

Each fixture under evals/<name>/ ships a prompts.json with regex assertions on the generated SOQL (mustInclude / mustNotInclude / exact object match). Brittle assertions get relaxed when real model behavior reveals equally-correct alternatives. The runner prints per-cell pass/fail with the generated SOQL on failure — diagnosis is immediate, not buried in stack traces.

Where the gaps are

What's still failing — and what each failure tells us.

Real grounding gaps (1 each, both models)

  • Picklist-value grounding. When the domain term lives only as a picklist value (Product2.Family = 'Flight') with no matching record type or api name, scoring finds nothing and the picker has no signal. Next strategy: score picklist values across describes during candidate selection.

Model-comprehension gaps (Haiku-specific)

  • Primary-noun confusion. "Cancelled flight with assignments" → Haiku picks Flight_Assignment__c instead of Flight__c (the parent). Could mitigate with a primary-vs-supporting object hint in the system prompt.
  • "Which X" answer-object rule. Haiku writes SELECT DISTINCT Account.Id FROM Case instead of FROM Account WHERE Id IN (...). The grammar nudge half-landed; tighter wording may close it.

Not a grounding problem

  • Haiku is structurally weaker at SOQL grammar than Sonnet. Some of these gaps may be cheaper to close by bumping the production default to Sonnet rather than over-engineering grounding for Haiku.

Summary

The bottom line.

Customers' Salesforce orgs are wildly different from each other. @soql now adapts to that variation by grounding against each org's actual metadata — record types, custom fields, picklists, record counts — instead of relying on training-data priors. Every change to that grounding is measured against a deterministic eval substrate that includes real customer-org snapshots, so we ship on data, not vibes.
