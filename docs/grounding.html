How @soql grounding works

Why "get me a cancelled flight" produces different SOQL in different orgs — and which signals the model uses to figure that out.

What problem are we solving?

The same natural-language prompt should produce different queries against different orgs, because the same business concept gets modeled differently:

  • One org calls a flight a Flight__c custom object.
  • Another models it as a Product2 record with Family = 'Flight'.
  • A third uses Asset with a Flight_Assignment record type to represent the assignment of a flight to a customer.
  • A Data Cloud install adds Flight_Assignments__dlm alongside, mirroring the operational object — same words, different system.

Claude can't guess which of those an org uses from training-data priors. The answer lives in the org's metadata. Grounding is the act of pulling the right slice of that metadata into the prompt so the model has the signal it needs to pick the right object and the right fields.

What signals exist in a Salesforce org?

Five sources of grounding signal, each one HTTP call away:

SignalAPI endpointWhat it gives you
describeGlobal GET /sobjects Every queryable object's api name + label + key prefix. Cheap, cached.
Per-object describe GET /sobjects/<Name>/describe Fields, types, references, picklist values, record-type metadata for one object. Moderately expensive per call.
Record types SELECT FROM RecordType One row per record type org-wide: SobjectType, DeveloperName, Name. Where business domain terms live on standard objects.
Custom field metadata Tooling: EntityDefinition / CustomField Field labels + help text across the org. Heavier — thousands of rows in mature orgs.
Record counts SELECT COUNT() FROM <Name> Per-object usage. Tiebreaker when objects look similar — the populated one is usually the operational one.

What @soql does today

prompt: "cancelled flights with assignments" | v [1] describeGlobal (cached) <-- only api name + label per object | v [2] Lexical scoring <-- tokenize prompt, regex-match | against each object's label | and api name v [3] Top 1-3 candidates | v [4] describe + COUNT() in parallel <-- pulls fields, picklists, | references, record counts v [5] Send schema + counts to Claude | v [6] Claude returns SOQL JSON | v [7] /query?explain validates <-- planner is ground truth; | on error, retry with | cumulative error context v result

The lexical scorer at step [2] is the bottleneck. It only ever sees api names and labels — nothing about record types, custom fields, picklist values, or what kind of records the org actually contains.

Where the current approach breaks

FAILURE 1  Lexical lure

Data Cloud objects (__dlm, __dll, __cio, __chn, …) ride alongside operational objects and frequently carry labels that mirror them exactly. The DMO Flight_Assignments__dlm labeled "Flight Assignments" scores higher on the prompt "flight assignments" than the real Asset object does — because Asset's label is just "Asset".

Mitigation: the feature/soql-grounding branch filters Data Cloud suffixes out of the candidate pool entirely. Necessary, but not sufficient — see Failure 2.

FAILURE 2  No lexical signal

When the prompt's domain terms don't appear in any object's label or api name, scoring returns zero candidates. The code falls back to a "picker" mode where Claude is handed the bare list of every object's api name + label and asked to pick one — no fields, no record types, no counts. Claude then guesses from abstract pattern alone.

Concrete: in an aviation org where flights are Product2 records and assignments are Asset records, the prompt "cancelled flight that has flight assignments" produces this SOQL after we filter out the DMO: 

SELECT Id, OrderNumber, Status, (SELECT Id FROM OrderItems)
FROM Order WHERE Status = 'Cancelled' LIMIT 1

Well-formed. Wrong object. Claude pattern-matched "cancelled X with sub-items" to Order + OrderItems, because that's the textbook Salesforce shape and nothing in the candidate list told it otherwise.

What richer grounding looks like

Three strategies, roughly in increasing cost and power. Each one surfaces a different slice of the org metadata into either scoring or the prompt itself.

1. Record-type-aware scoring  cheap, broad win

One additional SOQL query, cached for ~30 min: 

SELECT SobjectType, DeveloperName, Name
FROM RecordType
WHERE IsActive = true

Returns a few hundred rows at most. In the aviation fixture above, you'd get: 

Asset    / Flight_Assignment / "Flight Assignment"
Product2 / Flight            / "Flight"
Product2 / Aircraft          / "Aircraft"
Case     / Customer_Complaint / "Customer Complaint"
…

In step [2] of the flow, the scorer also iterates over record types and adds their score to the parent object. "Flight" lexically attaches to Product2 + Asset; "flight assignment" attaches to Asset specifically. Suddenly both real objects outscore everything else, get their describes fetched, and Claude has the schema it needs.

Bonus: the relevant record-type developer names get included in the prompt's schema dump, so Claude can emit RecordType.DeveloperName = 'Flight_Assignment' as a filter.

2. Picker fallback enrichment  additive

Even when scoring still finds nothing, the picker fallback (step where Claude picks from the bare object list) can be given richer signal — record types per object, populated picklist values for a curated set of "usually customized" standards (Product2.Family, Case.Type, Account.Industry), top custom field labels per object. Same prompt, better grounding, no new API calls beyond what (1) already fetched.

3. Custom field label scoring  expensive, last-resort

Tooling-API query against EntityDefinition / CustomField for label and help-text per field across the org. Returns thousands of rows in mature orgs — needs careful caching and probably a relevance filter (skip fields named "Date", "Notes" etc. that have no domain content). Score field labels against the prompt, propagate to their parent objects.

Best signal for orgs where the domain term lives only in field names — e.g. Product2.Flight_Number__c in an org that never bothered to set up a Flight record type. Add when (1) and (2) leave clear gaps.

Where validation fits in

None of the above changes the validator. Step [7] of the flow — pinging /query?explain=<soql> — remains the authoritative check. The Salesforce planner refuses to parse invalid SOQL and returns a diagnostic message, which we feed back to Claude for self-correction.

Grounding improves the object choice and the field references — making the planner's "object 'Foo' is not supported" or "no such column 'Bar'" errors much less common. The retry loop catches what slips through.

How we measure it

The eval harness in test/soql-eval.js reproduces arbitrary org shapes from fixtures — sobjects.json, per-object describes, counts, record types — and runs generateSoql() against real Anthropic calls. Each fixture × prompt cell asserts on the generated SOQL (object name, included/excluded patterns).

The harness is intentionally separate from the grounding work: it gives every proposed strategy a number to be measured against, before and after. 

ANTHROPIC_API_KEY=sk-ant-... npm run eval:soql

Current baseline against main: 0/1 passing — Data Cloud DMO wins. With the Data Cloud filter alone (feature/soql-grounding): still 0/1 — Claude defaults to Order for lack of grounding signal. Record-type scoring is the next experiment.
